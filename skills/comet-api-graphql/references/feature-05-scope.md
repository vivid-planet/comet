# Feature: Scoped Entities

| Name                     | Description                                                                                                                                                   |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Embedded scope field** | Entity owns its scope as a grouped `@Embeddable()` object stored in a `scope` field                                                                           |
| **Flat scope fields**    | Entity owns its scope as plain `@Property` fields directly on the entity (e.g. `domain`, `language`)                                                          |
| **Async via relation**   | Entity derives its scope from a related entity — either by accessing the relation's FK (`.id`) directly, or by loading the related entity with `loadOrFail()` |
| **No scope**             | Entity is not scoped — `skipScopeCheck: true`                                                                                                                 |
| **Injectable service**   | Complex scope resolution via a NestJS service with DI — used for page tree documents (`PageTreeNodeDocumentEntityScopeService`). Not generated by this skill  |

## Choosing a scope pattern

**Embedded scope field** is typical when the entity's content exists independently per scope — for example, a news article that exists separately for each domain and language. The API exposes a `scope` argument to select which version to read or write.

**Flat scope fields** appears on entities that carry `domain` and `language` as direct properties rather than a grouped `scope` object.

**Async via relation — FK only** fits entities that belong to a specific business entity acting as the scope anchor (e.g. a product belonging to a dealer). The entity is not duplicated per scope — it simply lives under one scope anchor. The API does not use a `scope` argument; the relation ID is passed directly instead.

**Async via relation — loadOrFail** fits sub-entities that do not carry scope themselves and need to derive it by loading their parent (e.g. a comment on a news article that itself has an embedded scope).

**No scope** is for entities that are truly global and not tied to any scope dimension.

> **Never use `getEntity()`** in a `@ScopedEntity` callback on a `Ref<T>` — it only works when already loaded in the identity map, which is not guaranteed. Use `loadOrFail()` instead.

## Embedded scope field

Entity owns its scope as a grouped `@Embeddable()` object stored in a field named `scope`. The `@Embeddable()` class serves as both GraphQL ObjectType and InputType — **no separate scope input DTO file is needed**.

> Check `app.module.ts` for the project's `ContentScope` shape to match the fields of the embeddable.

### Entity

```typescript
@Embeddable()
@ObjectType("")
@InputType("NewsContentScopeInput")
export class NewsContentScope {
    @Property({ columnType: "text" })
    @Field()
    @IsString()
    domain: string;

    @Property({ columnType: "text" })
    @Field()
    @IsString()
    language: string;
}

@Entity()
@ObjectType()
@ScopedEntity((news) => news.scope)
export class News extends BaseEntity {
    // ...
    @Embedded(() => NewsContentScope)
    @Field(() => NewsContentScope)
    scope: NewsContentScope;
    // ...
}
```

> If the embedded type doesn't match `ContentScope` exactly, extract explicitly: `(entity) => ({ domain: entity.scope.domain, language: entity.scope.language })`.

### @RequiredPermission — NO skipScopeCheck

```typescript
@RequiredPermission(["news"])
```

### Args DTO — scope as required first field

```typescript
@ArgsType()
export class NewsListArgs extends OffsetBasedPaginationArgs {
    @Field(() => NewsContentScope)
    @ValidateNested()
    @Type(() => NewsContentScope)
    scope: NewsContentScope;
    // ...
}
```

### Service deltas

- `findAll`: `where.scope = scope` — assign as object, **do NOT spread**
- `create`: pass `scope` as a named field in `entityManager.create()`, **do NOT spread**
- `findBySlug` (if applicable): `{ slug, scope }` — no spread

### Resolver deltas

- `createNews` mutation takes `@Args("scope", { type: () => NewsContentScope }) scope: NewsContentScope` as first arg
- `newsBySlug` query (if applicable) takes `@Args("scope", { type: () => NewsContentScope }) scope: NewsContentScope`

## Flat scope fields

Entity owns its scope as plain `@Property` fields directly on the entity. A separate scope input DTO is needed because the fields are not an `@Embeddable()`.

> Check `app.module.ts` for the project's `ContentScope` shape to match the fields.

### Scope Input DTO — `dto/{entity-name}-scope.input.ts`

```typescript
@ObjectType()
@InputType("NewsScopeInput")
export class NewsScope {
    @Field()
    @IsString()
    domain: string;

    @Field()
    @IsString()
    language: string;
}
```

> **Naming**: `{EntityName}Scope` / `"{EntityName}ScopeInput"`.

### Entity

```typescript
@Entity()
@ObjectType()
@ScopedEntity((news) => ({
    domain: news.domain,
    language: news.language,
}))
export class News extends BaseEntity {
    // ...
    @Property({ type: "text" })
    @Field()
    @IsString()
    domain: string;

    @Property({ type: "text" })
    @Field()
    @IsString()
    language: string;
    // ...
}
```

### @RequiredPermission — NO skipScopeCheck

```typescript
@RequiredPermission(["news"])
```

### Args DTO — scope as required first field

```typescript
@ArgsType()
export class NewsListArgs extends OffsetBasedPaginationArgs {
    @Field(() => NewsScope)
    @ValidateNested()
    @Type(() => NewsScope)
    scope: NewsScope;
    // ...
}
```

### Service deltas

- `findAll`: `Object.assign(where, scope)` — spread flat scope fields into `where`
- `create`: `...scope` spread into `entityManager.create()`
- `findBySlug` (if applicable): `{ slug, ...scope }` — spread scope fields

### Resolver deltas

- `createNews` mutation takes `@Args("scope", { type: () => NewsScope }) scope: NewsScope` as first arg
- `newsBySlug` query (if applicable) takes `@Args("scope", { type: () => NewsScope }) scope: NewsScope`

## Async via relation

Entity derives its scope from a related entity. There is **no `scope` arg** in the API — the related entity's ID is passed as a dedicated arg (see [feature-03-dedicated-arg.md](feature-03-dedicated-arg.md)).

### Sub-case: FK only — scope = the relation's ID

Access `.id` directly on the `Ref<T>` — no async load required:

```typescript
@Entity()
@ObjectType()
@ScopedEntity((product) => ({
    dealer: product.dealer.id,
}))
export class Product extends BaseEntity {
    // ...
    @ManyToOne(() => Dealer, { ref: true })
    dealer: Ref<Dealer>;
    // ...
}
```

### Sub-case: loadOrFail — scope needs properties of the related entity

```typescript
@Entity()
@ObjectType()
@ScopedEntity(async (comment) => {
    const news = await comment.news.loadOrFail();
    return { domain: news.scope.domain, language: news.scope.language };
})
export class NewsComment extends BaseEntity {
    // ...
    @ManyToOne(() => News, { ref: true })
    news: Ref<News>;
    // ...
}
```

### @RequiredPermission — NO skipScopeCheck

```typescript
@RequiredPermission(["newsComment"])
```

### Args DTO — relation ID as required first field, no scope field

```typescript
@ArgsType()
export class NewsCommentListArgs extends OffsetBasedPaginationArgs {
    @Field(() => ID)
    newsId: string;
    // ...
}
```

### Service deltas

- `findAll` takes `newsId: string` as first param: `where.news = newsId`
- `create` takes `newsId: string` as first param: load the related entity, then `Reference.create(relatedEntity)`

### Resolver deltas

- No `scope` arg anywhere
- `newsCommentList` query and `createNewsComment` mutation both take `@Args("newsId", { type: () => ID }) newsId: string` as first arg
- Both decorated with `@AffectedEntity(News, { idArg: "newsId" })`

---

## No scope

Entity has no scope and no `@ScopedEntity`. Use `skipScopeCheck: true` on `@RequiredPermission`.

```typescript
@RequiredPermission(["products"], { skipScopeCheck: true })
```

No further changes. Service and resolver follow the base patterns without any scope handling.
